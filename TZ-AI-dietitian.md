# Техническое задание: AI-диетолог

**Версия документа:** 0.1  
**Статус:** черновик (дизайн и визуальные макеты — отдельной поставкой позже)

---

## 1. Цель продукта

Веб-приложение **AI-диетолог** помогает пользователю:

- вести учёт питания и привычек;
- получать персональные рекомендации по рациону с учётом целей (похудение, набор, поддержание, здоровье), ограничений и предпочтений;
- задавать вопросы в свободной форме и получать ответы, опирающиеся на **контекст пользователя** и **базу знаний** (RAG), а не только на общие знания модели.

Продукт **не заменяет** очную консультацию врача/диетолога: в ТЗ закладываются дисклеймеры и границы ответственности.

---

## 2. Ключевые технологии (обязательные)

| Компонент | Технология | Назначение |
|-----------|------------|------------|
| ORM и миграции | **Prisma** | схема БД, миграции, типобезопасный доступ к данным |
| База данных + векторный поиск | **PostgreSQL** + расширение **pgvector** («vec SQL») | хранение профилей, дневников, документов базы знаний; семантический поиск по эмбеддингам для RAG |
| LLM и маршрутизация моделей | **OpenRouter** (API-ключ передаётся отдельно, не хранится в репозитории) | единая точка доступа к разным моделям (чат, суммаризация, классификация намерений) |

## 3. Подобранный остальной стек

### 3.1. Клиент

- **React 19** + **TypeScript** (`strict: true`)
- **Vite** — сборка и dev-сервер
- **React Router** — маршрутизация
- **TanStack Query** — серверное состояние, кэш, повторы запросов
- **Zod** — валидация форм и ответов API на клиенте
- Стили: **Tailwind CSS** + headless-компоненты (**Radix UI**) до появления финального дизайна; темing подготовить под будущую дизайн-систему

### 3.2. Сервер

- **Node.js** (LTS) + **Fastify** — HTTP API, валидация входа, плагины
- **Prisma Client** — работа с PostgreSQL/pgvector
- Вызовы OpenRouter: **нативный `fetch`** или лёгкий HTTP-клиент (без тяжёлых SDK при необходимости)

### 3.3. Эмбеддинги и RAG

- Генерация эмбеддингов для чанков базы знаний и (опционально) для пользовательских заметок — через **OpenRouter** (модель с поддержкой embeddings) или отдельный endpoint, зафиксированный в конфиге
- Хранение векторов в PostgreSQL через **pgvector**; поиск **cosine similarity** / **L2** — по согласованию в настройках

### 3.4. Инфраструктура и качество

- **Docker Compose** — локально: PostgreSQL с pgvector, (опционально) pgAdmin
- **ESLint** + **Prettier** + **Husky** + **lint-staged**
- Переменные окружения: **dotenv** / встроенная загрузка в runtime; секреты только из env (в т.ч. `OPENROUTER_API_KEY`)

### 3.5. Деплой (ориентир)

- Контейнеризация приложения (Docker); БД — управляемый PostgreSQL с поддержкой pgvector (Supabase, Neon, Railway, собственный VPS — уточняется на этапе внедрения)

## 4. Функциональные требования

### 4.1. Пользователь и доступ

- Регистрация / вход (минимум: email + пароль; расширение через OAuth — по желанию в следующих итерациях)
- Сессии: JWT в httpOnly cookie **или** server-side sessions в БД — выбрать один подход и зафиксировать в реализации
- Профиль: базовые антропометрические и целевые данные (возраст, пол, вес, рост, цель, активность, аллергии/непереносимости — перечень полей уточняется при проектировании схемы)

### 4.2. Дневник питания

- Приём пищи: время, описание блюда (текст), оценка порции
- Опционально: разбор текста через LLM в структурированный JSON (ккал, БЖУ) с явной пометкой «оценка модели, не лабораторный анализ»
- История по дням, фильтры, экспорт (CSV) — низкий приоритет, backlog

### 4.3. Чат с AI-диетологом

- Диалог с сохранением истории в БД
- Перед ответом: retrieval из **pgvector** по релевантным чанкам (политики питания, FAQ продукта, санитарные ограничения для ответов)
- Системный промпт: роль диетолога-ассистента, запрет на диагнозы и назначение лекарств, рекомендация обратиться к врачу при острых симптомах

### 4.4. База знаний (админ / контент)

- CRUD для «документов» и чанков (или загрузка файла → разбиение на чанки → эмбеддинги)
- Пересчёт эмбеддингов при смене модели — отдельная задача/скрипт

## 5. Модель данных (черновик для Prisma)

Сущности (уточняются в `schema.prisma`):

- `User`, `Session` / refresh-токены (если JWT — упрощённо)
- `Profile` (1:1 к пользователю)
- `MealEntry` (привязка к пользователю и дате)
- `ChatThread`, `ChatMessage`
- `KnowledgeDocument`, `KnowledgeChunk` с полем вектора типа, поддерживаемого Prisma + pgvector (например `Unsupported("vector(1536)")` или актуальный синтаксис на момент реализации)

Индексы для векторного поля — по документации pgvector и Prisma.

## 6. API (высокоуровнево)

REST (или tRPC — по решению команды; в ТЗ зафиксирован REST как дефолт для простоты интеграции):

- `POST /auth/*` — регистрация, вход, выход
- `GET/PATCH /profile`
- `GET/POST /meals`, `GET /meals?date=`
- `GET/POST /chat/threads`, `POST /chat/threads/:id/messages`
- Админ: `POST /admin/knowledge/documents` (защита ролью)

Все ответы с единым форматом ошибок; rate limiting на эндпоинты с LLM.

## 7. Нефункциональные требования

- **Безопасность:** хеширование паролей (argon2/bcrypt), HTTPS в проде, CORS по белому списку, лимиты на размер тела запроса
- **Наблюдаемость:** структурированные логи (pino), корреляционный id запроса
- **Производительность:** пагинация списков; для чата — лимит истории, передаваемой в LLM (скользящее окно + summary при необходимости — backlog)
- **Конфиденциальность:** минимизация ПДн в промптах; политика хранения и удаления данных (GDPR-ориентир — по запросу)

|------------|----------|

## 10. Критерии приёмки MVP

1. Пользователь может зарегистрироваться, войти, заполнить профиль.  
2. Пользователь может добавить приём пищи и видеть список за день.  
3. Пользователь может вести чат; ответы учитывают retrieved-чанки из pgvector при наличии загруженной базы знаний.  
4. Prisma-миграции применяются на чистую БД без ручных правок.  
5. Ключ OpenRouter читается только из окружения; в репозитории нет секретов.
